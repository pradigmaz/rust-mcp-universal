# Визуальные Графы И Таблицы Поверх MCP

## Короткий вывод

Текущий формат `rust-mcp-universal` уже подходит не только для CLI и агентского retrieval, но и для человеческих визуальных поверхностей. Ограничение не в MCP, а в том, что графы и таблицы должны строиться не "из всего подряд", а из стабильных структурированных JSON-ответов: `workspace_brief`, `rule_violations`, `related_files_v2`, `symbol_lookup_v2`, `symbol_references_v2`, `call_path`, `query_report`, `context_pack`.

Правильная модель: MCP остается backend/data-provider, а визуализация живет в отдельном UI-слое. Не консольном. Лучший путь: web UI или desktop-pane, который запрашивает данные у MCP и рендерит их как интерактивные графы, таблицы, карточки и heatmap-панели.

## Уточненная цель

Главная цель визуального слоя не в том, чтобы "рисовать связи", а в том, чтобы вскрывать расхождение между внешне рабочей системой и ее внутренней архитектурой.

Типичный случай:

- продукт "работает";
- по API и UI все выглядит нормально;
- но внутри модули сцеплены хаотично;
- зависимости идут не там, где ожидаются;
- один и тот же кусок логики протянут через несколько несвязанных зон;
- hotspot-файлы управляют слишком многим сразу.

Значит, визуализация должна отвечать не на вопрос "что с чем связано вообще", а на более полезные вопросы:

- где реальные центры управления проектом;
- где скрытые god-modules;
- где есть подозрительная связность;
- где архитектура не совпадает с ожидаемой;
- где quality-проблемы накладываются на graph-проблемы.

Это же относится и к таблицам: они нужны не просто как список нарушений, а как панель инженерных hotspot-мест.

## Что Уже Есть Как Основа

- `workspace_brief` уже умеет давать краткую картину по репозиторию, языкам, символам и quality summary.
- `rule_violations` уже умеет возвращать machine-readable нарушения, метрики, сортировку, статус и location.
- навигационные `*_v2` tools уже дают связи по символам и файлам.
- `call_path` уже дает bounded graph-path между двумя точками.
- `query_report` уже дает объяснение retrieval pipeline.

Это значит, что база для "человеческого слоя" уже есть. Не хватает не данных как класса, а удобного представления и пары специализированных MCP-endpoint'ов.

## Что Имеет Смысл Визуализировать

### 1. Граф связей репозитория

Obsidian-style, но не "космос ради космоса". Нужны 3 режима:

- `directory graph`: директории и крупные модули как узлы;
- `file dependency graph`: файлы и их связи;
- `symbol graph`: функции, типы, модули, вызовы и references.

Но практически полезнее думать не в терминах "один граф", а в терминах нескольких картинок на разные вопросы:

- `expected architecture graph`: как проект должен быть организован по директориям/зонам;
- `actual dependency graph`: как он реально связан;
- `hotspot overlay graph`: где на эти связи накладываются quality-нарушения;
- `hub graph`: какие узлы имеют слишком большой fan-in/fan-out;
- `cycle graph`: где есть циклы и псевдо-модульность.

Узлы должны иметь:

- тип;
- имя;
- путь;
- размер;
- language;
- score/weight;
- количество нарушений;
- severity color;
- краткий hover summary.

Ребра должны иметь:

- тип связи: `calls`, `references`, `imports`, `depends_on`, `retrieval_neighbor`;
- силу/вес;
- направление;
- источник, по которому связь была построена.

Отдельно важно: граф должен уметь показывать "подозрительность" связи. Не только факт, что связь есть, но и почему она выглядит плохой:

- межслойная связь, которую не ожидали;
- слишком много исходящих зависимостей;
- слишком много входящих зависимостей;
- связь через oversized/god-file;
- связь через узел с большим числом нарушений.

### 2. Таблицы для человека

Нужны не сырые JSON, а нормальные human tables:

- таблица quality violations;
- таблица top hot-spots;
- таблица "новые нарушения";
- таблица "самые связанные узлы";
- таблица "самые подозрительные модули".

Для твоего сценария я бы добавил еще 4 обязательные таблицы:

- `oversized_files`
- `deep_nesting_hotspots`
- `architectural_hubs`
- `layer_breaks`

Для violations таблицы минимальный набор колонок:

- `path`
- `rule_id`
- `actual`
- `threshold`
- `severity`
- `language`
- `location`
- `quality_mode`
- `suggested_action`

Для hotspot-таблицы полезные колонки:

- `path`
- `size_bytes`
- `non_empty_lines`
- `max_line_length`
- `import_count`
- `graph_edge_out_count`
- `symbol_count`
- `violation_count`
- `risk_score`

Для structural smell-таблиц стоит отдельно показывать:

- слишком большие файлы;
- слишком длинные функции;
- слишком глубокую вложенность;
- слишком большой fan-out;
- слишком большой fan-in;
- циклы;
- нарушение ожидаемого layer boundary.

### 3. Карточки и дашборды

Нужны компактные summary surfaces:

- status card по репозиторию;
- quality card;
- retrieval card;
- hotspots card;
- "что изменилось после refresh" card.

## Как Это Должно Выглядеть

### Главный экран

Не терминал и не dump JSON. Экран должен быть похож на аналитическую IDE-панель:

- слева фильтры и режимы графа;
- в центре интерактивный canvas-граф;
- справа detail panel выбранного узла;
- снизу переключаемая таблица.

### Поведение

- zoom/pan как в Obsidian;
- cluster collapse/expand;
- клик по узлу открывает связи и таблицу связанных нарушений;
- hover показывает summary;
- двойной клик уводит к файлу/символу/карте связей;
- переключатель layout: `force`, `hierarchy`, `radial`.

### Цветовая схема

- нейтральные узлы: серо-синие;
- quality-hot nodes: желтый/оранжевый/красный;
- retrieval-selected nodes: синий акцент;
- deleted/stale/degraded состояния: отдельный статусный бейдж.

Цвета должны объяснять риск, а не украшать экран.

## Какие MCP Surface Стоит Добавить

### 1. `visual_graph`

Возвращает готовый graph payload:

- `nodes`
- `edges`
- `clusters`
- `legend`
- `summary`

Параметры:

- `graph_type`: `directory`, `file_dependency`, `symbol`, `retrieval`, `quality`
- `center_path`
- `center_symbol`
- `depth`
- `limit`
- `include_quality`
- `include_metrics`

### 2. `visual_table`

Возвращает human-table payload:

- `columns`
- `rows`
- `sort`
- `filters`
- `summary`

Типы таблиц:

- `violations`
- `hotspots`
- `new_violations`
- `graph_hubs`
- `retrieval_candidates`

### 3. `visual_dashboard`

Агрегированный ответ для главного UI-экрана:

- repo summary
- quality summary
- top hot files
- top hot rules
- graph preview
- recent changes

### 4. `visual_export`

Генерация человекочитаемых артефактов:

- `html`
- `markdown`
- `json`
- позже возможно `png`/`svg`

Здесь важно: export не должен быть "консольным". HTML-экспорт — хороший baseline.

## Какой UI Лучше Делать

### Рекомендуемый вариант

Отдельный web UI, который общается с MCP-сервером.

Почему это лучше:

- графы удобнее рендерить в браузере;
- таблицы удобнее делать через нормальный grid;
- можно быстро сделать hover, filters, collapse, side panels;
- UI можно открыть и как standalone, и как embedded panel.

Технически для UI подходят:

- `React`
- `React Flow` или `Cytoscape.js`
- `TanStack Table`
- `ECharts` или `Recharts` для summary-графиков

### Менее удачный вариант

Генерировать только Mermaid или только markdown-графы.

Это полезно как экспорт, но недостаточно как основной UX. Mermaid хорош для report/export, но не как главный interactive layer.

## Что Реально Стоит Делать Первым

### Этап 1

- `visual_table` для нарушений и hotspots;
- `visual_dashboard` с quality summary;
- HTML-страница или web panel.

Это даст быструю пользовательскую ценность с минимальным риском.

### Этап 2

- `visual_graph` для directory/file graphs;
- фильтры по language, severity, rule, path prefix;
- click-through из графа в таблицу и обратно.

Именно на этом этапе появляется реальная польза для поиска "внутри всё криво":

- overlay quality на graph;
- выделение hub-узлов;
- подсветка подозрительных межмодульных связей;
- быстрый переход из "красного" узла в таблицу причин.

### Этап 3

- symbol graph;
- retrieval graph;
- saved views;
- diff mode: "что изменилось между двумя refresh".

## Что Плохо Подходит Под Формат

Следующее лучше не тащить в первую реализацию:

- LLM-judge графы без стабильной объяснимости;
- "красивые", но бесполезные космические force-layout'ы;
- все возможные связи сразу в одном экране;
- markdown/text-heavy анализ как основной graph source;
- автогенерация UI целиком из произвольного JSON без фиксированного контракта.

## Практическая Рекомендация

Лучшее следующее действие:

1. закрепить визуальный контракт как новые MCP tools `visual_table`, `visual_graph`, `visual_dashboard`;
2. сделать web UI поверх этих контрактов;
3. первым продактовым экраном сделать "Quality + Hotspots Dashboard";
4. вторым — "Dependency/Relation Graph";
5. экспорт оставить как HTML и Markdown summary, а не как primary UX.

Итог: это хорошо подходит под MCP. Но UI должен быть отдельным человеком-ориентированным слоем, а не попыткой превратить текущие CLI/MCP ответы в "псевдо-визуализацию" прямо в консоли.
